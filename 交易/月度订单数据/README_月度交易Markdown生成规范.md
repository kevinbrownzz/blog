## 2. 下次需要提供哪些数据

至少需要提供以下两类数据之一。可以只提供一个平台，也可以同时提供多个平台。

### 2.1 Binance 成交数据

推荐提供 JSON 数组，每一项是一笔真实成交撮合，而不是委托单。

必须包含：

| 字段 | 类型 | 含义 |
|---|---|---|
| `symbol` | 字符串 | 标的，例如 `BTCUSDT`、`NVDAUSDT` |
| `id` | 字符串或整数 | 成交 ID，同一撮合唯一 |
| `orderId` | 字符串或整数 | 订单 ID，同一订单可拆成多笔成交 |
| `side` | 字符串 | `BUY` 或 `SELL` |
| `price` | 数字或数字字符串 | 成交价格 |
| `qty` | 数字或数字字符串 | 成交数量 |
| `quoteQty` | 数字或数字字符串 | 成交额 |
| `realizedPnl` | 数字或数字字符串 | 已实现盈亏 |
| `commission` | 数字或数字字符串 | 手续费 |
| `commissionAsset` | 字符串 | 手续费币种，通常为 `USDT` |
| `time` | 整数 | Unix 毫秒时间戳 |
| `positionSide` | 字符串 | 通常为 `BOTH`、`LONG` 或 `SHORT` |
| `maker` | 布尔值 | 是否为 Maker 成交 |

推荐同时保留：

- `buyer`
- 原始导出文件名
- 账户或子账户标识


## 4. 仓位归并规则

### 4.1 一个仓位的定义

不能把一个订单等同于一个仓位。

同一平台、同一标的的净持仓：

1. 从 `0` 变为非 `0`，仓位开始；
2. 中间所有分批开仓、加仓、减仓和分批平仓均归入同一仓位；
3. 净持仓重新回到 `0`，仓位结束；
4. 如果一次成交从多仓直接反转为空仓，必须拆成：
   - 原多仓的平仓部分；
   - 新空仓的开仓部分；
5. Binance 和 Hype 即使交易同一标的，也必须分别建立仓位；
6. 同一仓位跨月时，保持同一个仓位 ID。

示例：

```text
BUY 0.20 BTC  → 开多
BUY 0.30 BTC  → 加多
SELL 0.10 BTC → 减多
SELL 0.40 BTC → 完全平多
```

以上全部属于同一个仓位。

---

### 4.2 仓位 ID 格式

平台前缀：

```text
Binance → MAIN
Hype    → HL
```

示例：

```text
MAIN-BTC-001
MAIN-NVDA-003
HL-MU-001
HL-NVDA-002
```

同一平台、同一标的按仓位开始时间顺序编号。

---

### 4.3 多空方向

- 净持仓大于0：多仓
- 净持仓小于0：空仓
- `BUY` 对多仓通常是开仓或加仓
- `SELL` 对多仓通常是减仓或平仓
- `SELL` 对空仓通常是开仓或加仓
- `BUY` 对空仓通常是减仓或平仓

必须根据成交前后的净持仓判断动作，不能只看 BUY/SELL。

## 5. 价格计算规则

### 5.1 平均开仓价

使用所有开仓和加仓成交的数量加权平均价：

```text
平均开仓价 = 开仓成交额合计 ÷ 开仓数量合计
```

### 5.2 平均平仓价

使用所有减仓和平仓成交的数量加权平均价：

```text
平均平仓价 = 平仓成交额合计 ÷ 平仓数量合计
```

### 5.3 未完全平仓

如果仓位尚未完全平仓：

- 平均开仓价仍正常计算；
- 平均平仓价只统计已完成的减仓和平仓部分；
- 若完全没有平仓成交，平均平仓价显示 `—`；
- 整仓净损益只显示目前已实现部分；
- 不得把未实现盈亏混入整仓净损益。

## 6. 月度损益和整仓损益

### 本月净损益

只统计该自然月内发生的成交所产生的：

```text
已实现盈亏 - 手续费
```

### 整仓净损益

统计同一仓位从开始到结束的完整生命周期净损益。

因此跨月仓位可能出现：

```text
本月净损益 ≠ 整仓净损益
```

月度表必须同时保留这两个字段。

## 14. 订单级分拆成交明细固定格式

每个仓位使用可折叠区域：

```markdown
<details>
<summary>展开本月 N 条订单级明细</summary>

| 时间 | 订单ID | 仓位动作 | 买卖方向 | 数量 | 均价 | 成交额（USDT） | 毛已实现盈亏 | 手续费 | 净已实现盈亏 | 撮合笔数 |
|---|---|---|---|---:|---:|---:|---:|---:|---:|---:|

</details>
```

仓位动作只能使用：

```text
开仓
加仓
减仓

```

同一交易所订单拆成多笔撮合时：

- 在原始成交中全部保留；
- 在订单级表格中按订单 ID 聚合；
- 聚合均价使用成交数量加权平均价；
- 撮合笔数显示该订单包含多少笔真实成交。

## 16. 交易总结的写法

每个月的“交易总结”不能只有表格，至少包括以下内容：

1. 本月成交记录、订单数、仓位数、成交额；
2. 毛已实现盈亏、手续费、净已实现盈亏；
3. 盈利贡献最大的三个标的；
4. 亏损拖累最大的三个标的；
5. 本月贡献最大的仓位；
6. 本月拖累最大的仓位；
7. Binance 和 Hype 各自的净贡献；
8. 手续费和交易频率是否过高；
9. 是否存在单笔大亏覆盖多笔小盈利；
10. 如果是未完整月份，明确说明统计截止时间。

不能只复述数字，还要解释交易结构。

## 19. 数据质量检查

生成前必须检查：

- 成交 ID 是否重复；
- 同一订单是否包含多笔撮合；
- 时间戳是否异常；
- 买卖方向是否缺失；
- 价格、数量、成交额是否为负数；
- 手续费币种是否一致；
- 净持仓是否出现无法解释的跳变；
- 是否存在文件开始前已有仓位；
- 数据末尾是否还有未平仓仓位；
- 跨月仓位是否保持同一个仓位 ID；
- 平均开仓价和平仓价是否按数量加权；
- 月度净损益合计是否等于总净损益；
- 平台汇总、标的汇总、仓位汇总是否能够相互核对。

如果文件开始时已有历史仓位，必须提示：

```text
当前仓位归并可能缺少更早的开仓成交，请补充更早数据。
```
